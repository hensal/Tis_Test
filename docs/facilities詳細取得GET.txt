facilities詳細取得GET
======================

Source checked:
- Excel file: APIテスト仕様書(facilities).xlsx
- Excel sheet: facilities詳細取得GET
- Code checked:
  - src/main/java/com/example/demo/controller/FacilitiesController.java
  - src/main/java/com/example/demo/service/FacilitiesService.java

Common Postman setup
--------------------
Endpoint base:
http://localhost:8081/facilities

Header:
Authorization: Bearer <access_token>

Use a SYS_ADMIN token for this Excel section unless the test case says otherwise.

Useful SQL: list facility IDs before testing
--------------------------------------------
Use this first to see all facility IDs in the DB:

SELECT facility_id, parent_id, facility_name, deleted_at
FROM facilities_tree
ORDER BY facility_id;

Use this to see only active facilities:

SELECT facility_id, parent_id, facility_name
FROM facilities_tree
WHERE deleted_at IS NULL
ORDER BY facility_id;

If your DB still uses old column names, use this instead:

SELECT facility_id, parent_id, node_name, deleted_at
FROM facilities_tree
ORDER BY facility_id;

Standard check pattern for every response item
----------------------------------------------
For each item in this document, use this order:
1. First find a facility_id that has the required test data in DB.
2. Send GET http://localhost:8081/facilities/<facility_id> in Postman.
3. Run the item-specific SQL.
4. Compare the SQL result with the JSON response field.

For example:
- images: first find a facility_id that has facility_images data.
- maps: first find a facility_id whose parent has maps data.
- map_points: first find a facility_id whose parent map has map_points data.
- roles: first find a facility_id that has facility_role_permission data.
- equipments: first find a facility_id that has equipments data.
- annotations: first find a facility_id that has annotations data.

Expected common error response:
{
  "success": false,
  "error_code": "invalid_request",
  "message": "リクエストパラメータが不正です",
  "errors": null,
  "data": null
}

1-1-5 facilityid=存在しないファシリティIDを指定
------------------------------------------------
Meaning:
Send a facility_id that does not exist in facilities_tree.

Before Postman, confirm the ID does not exist:
SELECT facility_id
FROM facilities_tree
WHERE facility_id = 999999999
  AND deleted_at IS NULL;

Expected DB result:
0 rows.

Postman:
GET http://localhost:8081/facilities/999999999

Expected by Excel:
A normal success response is returned with 0 records / no facility data.

Code check:
Current code calls FacilitiesService.getActiveFacility().
If no row exists, current code returns:
{
  "success": false,
  "error_code": "not_found",
  "message": "対象施設が見つかりません",
  "errors": null,
  "data": null
}

So this item is not fully aligned with the Excel yet if the Excel strictly requires success=true for 0 records.

1-1-6 facilityidが数値化できない文字列である
------------------------------------------------
Meaning:
Send a facility_id that cannot be converted to a number.

Postman:
GET http://localhost:8081/facilities/abc

Expected response:
The common invalid_request error response is returned.

Code check:
Current code checks !facilityId.matches("\\d+"), so this requirement is handled.

1-1-7 facilityidが19byteより多い文字列である
------------------------------------------------
Meaning:
Send a facility_id string longer than 19 bytes.
For ASCII numbers, 20 digits = 20 bytes, so this is enough for the test.

Postman:
GET http://localhost:8081/facilities/12345678901234567890

Expected response:
The common invalid_request error response is returned.

Code check:
Current code checks facilityId.getBytes(StandardCharsets.UTF_8).length > 19, so this requirement is handled.

1-1-8 {facilityId}/vr-view以外のサブパラメータが指定されている
----------------------------------------------------------------
Meaning:
Send a sub path other than the allowed vr-view path.

Postman:
GET http://localhost:8081/facilities/1/invalid-sub-path

Expected response:
The common invalid_request error response is returned.

Code check:
Current code has a catch-all endpoint:
GET /facilities/{facilityId}/{subPath}
It validates facilityId, then returns invalid_request. So unknown sub paths are handled.

Important note:
Current code also has:
GET /facilities/{facilityId}/view-html

If the Excel rule strictly means only /vr-view is allowed, then /view-html is a code/spec mismatch.
For this test case, use /invalid-sub-path, not /vr-view.

1-2-11 images
-------------
Meaning:
The response data.images list must match the DB result from FACILITY_IMAGES / facility_images.

  ft.facility_name AS test_facility_name,

-----------------
Meaning:
The response data.map_points list must match the DB result from MAP_POINTS / map_points.
For this test, use a facility where the related map has multiple map_point records.

First find a facility whose parent has map data and map points:

Simple count version:

SELECT
  child.facility_id AS test_facility_id,
  child.facility_name AS test_facility_name,

1-2-27 roles
------------
Meaning:
The response data.roles list must match the DB result from NODE_AUTHORITY_ROLE.
In the current PostgreSQL DB, NODE_AUTHORITY_ROLE is represented by facility_role_permission.
For this test, use a facility that has multiple facility_role_permission records.

First find a facility that has role permission data:

Simple count version:

SELECT
  ft.facility_id AS test_facility_id,
  ft.facility_name AS test_facility_name,
  COUNT(frp.facility_role_permission_id) AS role_count
FROM facilities_tree ft
JOIN facility_role_permission frp
  ON frp.facility_id = ft.facility_id
