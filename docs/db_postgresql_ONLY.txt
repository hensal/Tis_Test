PostgreSQL setup for Facilities API tests
=========================================

Purpose
-------
Use this setup on any computer to create the PostgreSQL tables and sample data
needed for the facilities Excel test cases.

Source documents used
---------------------
1. /Users/hensal/Desktop/JAVA_SPRING_BOOT/TEST_CASE/DB設計.md
2. /Users/hensal/Desktop/JAVA_SPRING_BOOT/TEST_CASE/サンプルデータ.md
3. /Users/hensal/Desktop/JAVA_SPRING_BOOT/TEST_CASE/sample_data.sql

Project files created for portable setup
----------------------------------------
1. docs/schema_DB設計_facilities.sql
   - Drops old app tables.
   - Creates tables based on DB設計.md.

2. docs/sample_data_DB設計.sql
   - Copied from the provided sample_data.sql.
   - Inserts the sample data described in サンプルデータ.md.

Important note
--------------
DB設計.md and sample_data.sql are not perfectly identical for role columns:

- DB設計.md describes keycloak_role_id.
- sample_data.sql inserts keycloak_role_name.

To make the setup runnable, docs/schema_DB設計_facilities.sql keeps both columns:

- keycloak_role_id
- keycloak_role_name

This is also practical because Keycloak tokens usually contain role names such as
SYS_ADMIN, R_USER01, viewer, etc.

Step 1: Create the database
---------------------------
In pgAdmin, connect to the default postgres database and run:

CREATE DATABASE vr_view_db;

Then right-click vr_view_db and open Query Tool for vr_view_db.

Step 2: Create all tables
-------------------------
Open and run this file in pgAdmin Query Tool:

docs/schema_DB設計_facilities.sql

If using psql:

psql -U postgres -d vr_view_db -f docs/schema_DB設計_facilities.sql

This recreates these tables:

- facilities_tree
- files
- permission_master
- facility_role_permission
- equipment_types
- equipment_masters
- equipments
- facility_images
- maps
- map_points
- hotspots
- annotation_type_master
- annotations
- annotation_comments
- annotation_files
- equipment_files

GET http://localhost:8081/facilities?keyword=東京

GET http://localhost:8081/facilities?parentId=1110

POST http://localhost:8081/facilities
Content-Type: application/json

Compatibility note with Java code
---------------------------------
This setup follows DB設計.md.

The Java facilities service should use these DB設計.md column names:

- facility_images.shooting_height_m / ceiling_height_m
  Old names. Do not use.

- facility_images.shooting_height / ceiling_height
  Correct DB設計.md names.

- facility_images.guid
  Old column. Do not use.

- map_points.target_type
  Old column. Do not use.

- equipments.content_id
  Old column. Do not use.

- equipments.equipment_id
  Correct DB設計.md name.

Quick troubleshooting
---------------------
If sample_data_DB設計.sql fails with a foreign key error:

1. Confirm schema_DB設計_facilities.sql completed successfully.
2. Confirm you are connected to vr_view_db, not postgres.
3. Re-run schema_DB設計_facilities.sql first because it drops and recreates the tables.
4. Then re-run sample_data_DB設計.sql.

If Postman returns permission_denied:

1. Get a fresh Keycloak access token.
2. For SYS_ADMIN tests, the token must include SYS_ADMIN.
3. For non-SYS_ADMIN tests, the role in the token must match a role in
   facility_role_permission, such as R_USER01 or R_PJMGR01.

If Postman returns SQL column errors:

The DB setup and Java code are using different column names. Use the
"Compatibility warning with current Java code" section above to decide whether
to change Java code or change the DB schema.
