Facilities Create API Test Steps: facilities新規作成(USER_ADMIN)
=================================================================

API:
POST http://localhost:8081/facilities

Common Postman setup:
1. Method: POST
2. URL: http://localhost:8081/facilities
3. Headers:
   Authorization: Bearer <USER_ADMIN access_token>
   Content-Type: application/json
4. Body:
   raw -> JSON

Expected common success response:
{
  "success": true,
  "error_code": null,
  "message": "ok",
  "data": {
    "facility_id": <created_facility_id>
  }
}

Expected common error response for invalid request parameters:
{
  "success": false,
  "error_code": "invalid_request",
  "message": "リクエストパラメータが不正です",
  "errors": null,
  "data": null
}

Expected common error response when USER_ADMIN does not have parent edit permission:
{
  "success": false,
  "error_code": "permission_denied",
  "message": "施設を登録する権限がありません",
  "errors": null,
  "data": null
}

Important:
- The login token must have USER_ADMIN or User_admin role.
- USER_ADMIN can call POST /facilities, but cannot create under any parent automatically.
- For USER_ADMIN, the user must also have edit permission for the parent facility.
- In this API, parent edit permission means:
  permission_id = 10
  facility_id = request parent_id
  keycloak_role_id = one of the login user's Keycloak roles
- If the token is missing or wrong, you will get invalid_token first.
- If the token does not have USER_ADMIN/User_admin/SYS_ADMIN, you will get permission_denied before DB logic.
- New test cases should always be appended at the bottom in Excel serial order.
- In SQL examples, <response_facility_id> means the actual data.facility_id returned by Postman.
  Example: if Postman returns "facility_id": 18, use 18 in the SQL.


Before Testing: Check Login User Roles
-------------------------------------

Call:
GET http://localhost:8081/api/keycloak/me

Headers:
Authorization: Bearer <USER_ADMIN access_token>

Confirm that the response data.roles contains the role you will use for DB permission.

Example:
{
  "roles": [
    "USER_ADMIN",
    "R_USER01"
  ]
}

In this example:
- USER_ADMIN allows calling POST /facilities.
- R_USER01 must be registered in facility_role_permission for the parent facility.


Before Testing: Add Parent Facility Edit Permission
--------------------------------------------------

Meaning:
When creating a child facility, USER_ADMIN must have edit permission on the parent facility.
This is the Excel condition:

SELECT COUNT(*)
FROM facility_role_permission
WHERE permission_id = 10
  AND facility_id = <parent_facility_id>
  AND keycloak_role_id IN (<login_user_roles>);

If this count is 1 or more, USER_ADMIN can create under that parent.
If this count is 0, the API should return permission_denied.

Example SQL:
If parent_id is 1110 and the login user has role R_USER01:

INSERT INTO facility_role_permission (
  permission_id,
  facility_id,
  keycloak_role_id,
  keycloak_role_name
)
VALUES (
  10,
  1110,
  'R_USER01',
  'R_USER01'
);

Check SQL:
SELECT COUNT(*) AS permission_count
FROM facility_role_permission
WHERE permission_id = 10
  AND facility_id = 1110
  AND keycloak_role_id IN ('R_USER01')
  AND deleted_at IS NULL;

Expected:
permission_count >= 1

1-5 リクエストパラメータ（最大のパラメータ送信：正常時）
------------------------------------------------------


1-5-1 事前準備：親施設の編集ロールを付与する
-------------------------------------------

Meaning:
Before USER_ADMIN creates a child facility, the login user must have edit permission for the parent facility.

Why:
USER_ADMIN can call POST /facilities, but creating under a parent facility requires permission_id = 10 on that parent.

SQL condition from Excel:
This count must be 1 or more.

SELECT COUNT(*) AS permission_count
FROM facility_role_permission
WHERE permission_id = 10
  AND facility_id = <facility_id_created_in_SYS_ADMIN_1_4_1>
  AND keycloak_role_id IN (<login_user_role_ids>)
  AND deleted_at IS NULL;

Example:
If the parent facility created in SYS_ADMIN 1-4-1 is 1110,
and the USER_ADMIN login user has role R_USER01:

INSERT INTO facility_role_permission (
  permission_id,
  facility_id,
  keycloak_role_id,
  keycloak_role_name
)
VALUES (
  10,
  1110,
  'R_USER01',
  'R_USER01'
);

Check:
SELECT COUNT(*) AS permission_count
FROM facility_role_permission
WHERE permission_id = 10
  AND facility_id = 1110
  AND keycloak_role_id IN ('R_USER01')
  AND deleted_at IS NULL;

Expected:
permission_count >= 1

If permission_count is 0:
POST /facilities should return:
{
  "success": false,
  "error_code": "permission_denied",
  "message": "施設を登録する権限がありません",
  "errors": null,
  "data": null
}

Common mistake:
The keycloak_role_id in facility_role_permission must match one of the roles inside the login user's access token.

Example:
If Keycloak user pjmgr01 has only this client role:
USER_ADMIN

Then this DB row is correct:
INSERT INTO facility_role_permission (
  permission_id,
  facility_id,
  keycloak_role_id,
  keycloak_role_name
)
VALUES (
  10,
  1110,
  'USER_ADMIN',
  'USER_ADMIN'
);

This DB row will not work unless the login user's token also contains R_USER01:
INSERT INTO facility_role_permission (
  permission_id,
  facility_id,
  keycloak_role_id,
  keycloak_role_name
)
VALUES (
  10,
  1110,
  'R_USER01',
  'R_USER01'
);

How to confirm the real login user's roles:
Call this API with the same token used for POST /facilities:

GET http://localhost:8081/api/keycloak/me

Check:
data.roles

The role in data.roles must match facility_role_permission.keycloak_role_id.

Important:
The roles array inside the POST /facilities request body is for permissions on the new facility.
It does not give the current login user permission to create under the parent.
