facilities一覧取得GETkeyword指定
=================================

1 List GET keyword search, SYS_ADMIN
====================================

Endpoint:
GET /facilities?keyword=<keyword>

User:
SYS_ADMIN access token

1-2 Response Parameters
-----------------------

1-2-1 Data count
----------------
Meaning:
All active facilities whose facility_name contains the keyword must be listed.

Step 1: choose a keyword that exists in facility_name.

Find example keywords:

SELECT facility_id, facility_name
FROM facilities_tree
WHERE deleted_at IS NULL
ORDER BY facility_id;

Step 2: check expected rows in DB.

SELECT *
FROM facilities_tree
WHERE facility_name LIKE '%<keyword>%'
  AND deleted_at IS NULL
ORDER BY sort_order ASC, facility_id ASC;

Step 3: test in Postman.

GET http://localhost:8081/facilities?keyword=<keyword>

Headers:
Authorization: Bearer <SYS_ADMIN access_token>

Expected:
- success = true
- data is an array
- data count matches the SQL result count
- rows are ordered by sort_order ASC, facility_id ASC

1-2-2 facility_id
-----------------
Check:
data[].facility_id must match facilities_tree.facility_id.

SQL:

  AND deleted_at IS NULL
ORDER BY sort_order ASC, facility_id ASC;

1-2-7 has_vr_image
------------------
Meaning:
For each facility in the response, has_vr_image is true when at least one active facility_images row exists.

SQL for one facility:

SELECT COUNT(*) AS vr_image_count
FROM facilities_tree ft
JOIN facility_images fi
  ON fi.facility_id = ft.facility_id
WHERE ft.facility_id = <facility_id>
  AND ft.deleted_at IS NULL
  AND fi.deleted_at IS NULL;

Expected:
- count >= 1: has_vr_image = true
- count = 0: has_vr_image = false

1-2-8 has_emergency_annotation
------------------------------
Meaning:
For each facility in the response, has_emergency_annotation is true when at least one active emergency annotation exists.
Current code treats emergency annotation_type as '10'.

SQL for one facility:

SELECT COUNT(*) AS emergency_annotation_count
FROM facilities_tree ft
JOIN annotations a
  ON a.facility_id = ft.facility_id
WHERE ft.facility_id = <facility_id>
  AND a.annotation_type = '10'
  AND ft.deleted_at IS NULL
  AND a.deleted_at IS NULL;

Expected:
- count >= 1: has_emergency_annotation = true
- count = 0: has_emergency_annotation = false

4 List GET keyword search, USER01
=================================

Endpoint:
GET /facilities?keyword=<keyword>

User:
USER01 access token

4-2 Response Parameters
-----------------------

4-2-1 Data count
----------------
Meaning:
For USER01, the API must return only active facilities that match the keyword and also have a facility role permission for one of USER01's roles.

In the Excel wording, NODE_AUTHORITY_ROLE means the same permission table used by this code:
facility_role_permission.

Step 1: get USER01 roles.

Postman:
GET http://localhost:8081/api/keycloak/me

Headers:
Authorization: Bearer <USER01 access_token>

Use the roles returned in data.roles.

Example:
USER01 role = R_USER01

Step 2: choose a keyword that should return data for USER01.

Find candidate facilities and roles:

SELECT
  ft.facility_id,
  ft.facility_name,
  ft.parent_id,
  frp.keycloak_role_id
FROM facilities_tree ft
JOIN facility_role_permission frp
  ON frp.facility_id = ft.facility_id
WHERE ft.deleted_at IS NULL
  AND frp.deleted_at IS NULL
ORDER BY ft.sort_order ASC, ft.facility_id ASC;

Step 3: check expected rows in DB.

Replace <keyword> and the role list with the roles from /api/keycloak/me.

SELECT DISTINCT ft.*
FROM facilities_tree ft
JOIN facility_role_permission frp
  ON frp.facility_id = ft.facility_id
WHERE ft.facility_name LIKE '%<keyword>%'
  AND ft.deleted_at IS NULL
  AND frp.keycloak_role_id IN ('R_USER01')
  AND frp.deleted_at IS NULL
ORDER BY ft.sort_order ASC, ft.facility_id ASC;

Expected:
- If the SQL returns N rows, the API data array must also contain N rows.
- If the SQL returns 0 rows, the API must return data: [].

Step 4: test in Postman.

GET http://localhost:8081/facilities?keyword=<keyword>

Headers:
Authorization: Bearer <USER01 access_token>

Expected:
- success = true
- data is an array
- data count matches the SQL result count
- rows are ordered by sort_order ASC, facility_id ASC

3 List GET keyword search, SYS_ADMIN, no matching data
======================================================

Endpoint:
GET /facilities?keyword=<keyword>

User:
SYS_ADMIN access token

3-1 Response Parameters
-----------------------

3-3-1 Data count
----------------
Meaning:
Use a keyword that does not match any active facility_name.
The API must return a normal response with 0 records.

Because this is a list API, 0 records should be returned as an empty array:

{
  "success": true,
  "error_code": null,
  "message": "ok",
  "data": []
}

Step 1: choose a no-match keyword.

Example keyword:

NO_MATCH_KEYWORD_999999

Step 2: confirm in DB that the keyword returns 0 records.

SELECT COUNT(*) AS facility_count
FROM facilities_tree
WHERE deleted_at IS NULL
  AND facility_name LIKE '%NO_MATCH_KEYWORD_999999%';

Expected:
facility_count = 0

Step 3: test in Postman.

Method:
GET

URL:
http://localhost:8081/facilities?keyword=NO_MATCH_KEYWORD_999999

Headers:
Authorization: Bearer <SYS_ADMIN access_token>

Expected response:

{
  "success": true,
  "error_code": null,
  "message": "ok",
  "data": []
}

Check:
- HTTP status is 200.
- success is true.
- error_code is null.
- data is an empty array.
