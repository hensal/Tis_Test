Facilities Create API Test Steps: facilities新規作成(SYS_ADMIN)
================================================================

API:
POST http://localhost:8081/facilities

Common Postman setup:
1. Method: POST
2. URL: http://localhost:8081/facilities
3. Headers:
   Authorization: Bearer <SYS_ADMIN access_token>
   Content-Type: application/json
4. Body:
   raw -> JSON

Expected common error response for invalid request parameters:
{
  "success": false,
  "error_code": "invalid_request",
  "message": "リクエストパラメータが不正です",
  "errors": null,
  "data": null
}

Important:
- The user token must have SYS_ADMIN role.
- If the token is missing or wrong, you will get invalid_token first.
- If the token does not have SYS_ADMIN, you will get permission_denied first.
- New test cases should always be appended at the bottom in Excel serial order.
- In SQL examples, <response_facility_id> means the actual data.facility_id returned by Postman.
  Example: if Postman returns "facility_id": 18, use 18 in the SQL.


1-1-5 facility_name文字列に制御コードを入れている
---------------------------------------------------

Meaning:
The facility_name contains a control character.
A control character is an invisible/non-printable character, for example:
- newline
- tab
- carriage return
- backspace

Why this should be rejected:
facility_name is a normal display name, so it should not contain invisible control codes.

Current code check:
FacilitiesService.requiredFacilityName(...)
It rejects facility_name when:
- it is blank
- it is more than 100 characters
- it contains Character.isISOControl(...)

Postman test body:
Use a newline escape sequence inside facility_name:

{
  "parent_id": 1,
  "facility_name": "会議室\nテスト",
  "public_flag": "02",
  "outer_flag": false,
  "facility_description": "control code test"
}

Alternative test body using tab:

{
  "parent_id": 1,
  "facility_name": "会議室\tテスト",
  "public_flag": "02",
  "outer_flag": false,
  "facility_description": "control code test"
}

Expected HTTP status:
400 Bad Request

Expected JSON response:
{
  "success": false,
  "error_code": "invalid_request",
  "message": "リクエストパラメータが不正です",
  "errors": null,
  "data": null
}

How to confirm DB was not changed:
Run this SQL before and after the Postman request.
The count should not increase.

SELECT COUNT(*) AS facility_count
FROM facilities_tree
WHERE deleted_at IS NULL;

If the API returns success, this test is failing.
If the API returns the invalid_request response above, this test is passing.

- tab: \t

If parent_id is sent as a normal JSON number, it cannot contain control characters.

1-4-1 DBの値 facility_id
-------------------------

Meaning:
After POST /facilities succeeds, the response facility_id must be the same ID that was inserted into the facilities_tree table.

Why this should be checked:
The API response must point to the actual database record created by the API.

Current code behavior:
FacilitiesService.createFacility(...)
The code inserts a row into facilities_tree using PostgreSQL auto-generated facility_id.
Then it returns that generated facility_id in the response.

Postman test:
Send a valid minimum create request:

{
  "facility_name": "会議室テスト",
  "parent_id": "",
  "target_facility_id": ""
}

Expected HTTP status:
200 OK

Example response:
{
  "success": true,
  "error_code": null,
  "message": "ok",
  "data": {
    "facility_id": 1001
  }
}

SQL check:
Replace <response_facility_id> with the facility_id returned by Postman.

SELECT facility_id, facility_name
FROM facilities_tree
WHERE facility_id = <response_facility_id>;

Expected SQL result:
- One row is returned.
- facility_id is exactly the same as the response data.facility_id.

Example:
If Postman returned:

{
  "data": {
    "facility_id": 1001
  }
}

Then run:

SELECT facility_id, facility_name
FROM facilities_tree
WHERE facility_id = 1001;

Passing condition:
The SQL returns one row with facility_id = 1001.

Failing condition:
- No row is returned.
- A different facility_id is returned.


1-4-2 DBの値 parent_id
-----------------------

Meaning:
For the created facility_id, the DB parent_id must match the parent_id sent in the request.

Why this should be checked:
The facility tree structure depends on parent_id.
If the API stores a different parent_id, the new facility will appear under the wrong parent.

Current code behavior:
FacilitiesService.createFacility(...)
The code parses parent_id from the request.
- If parent_id is "" or null, it stores NULL.
- If parent_id is a valid existing facility ID, it stores that ID.

Postman test body example 1: parent_id empty

{
  "facility_name": "会議室テスト",
  "parent_id": "",
  "target_facility_id": ""
}

Expected DB parent_id:
NULL

SQL check:
Replace <response_facility_id> with the facility_id returned by Postman.

SELECT facility_id, parent_id, facility_name
FROM facilities_tree
WHERE facility_id = <response_facility_id>;

Passing condition for example 1:
parent_id is NULL.

Postman test body example 2: parent_id = 1

{
  "facility_name": "会議室テスト",
  "parent_id": 1,
  "target_facility_id": ""
}

Expected DB parent_id:
1

SQL check:

SELECT facility_id, parent_id, facility_name
FROM facilities_tree
WHERE facility_id = <response_facility_id>;

Passing condition for example 2:
parent_id is 1.

Failing condition:
- parent_id is different from the request value.
- The created row does not exist.


1-4-14 DBの値 他の施設ツリーテーブルのレコード
------------------------------------------------

Meaning:
For the minimum create request, only one new facilities_tree row should be inserted.
No other facilities_tree rows should be inserted, updated, or deleted.

Excel condition:
この新規作成処理で、Insert、Update、Deleteされているレコードは無い
Insert、Delete：この新規作成処理前後でのレコード数が、新規作成レコードの＋１だけである
Update：updated_atが同じ日時のレコードが無い

Important:
This check is for the minimum request case where parent_id is empty/null.
If parent_id is set to an existing parent, the API may update the parent row's is_leaf and updated_at.
That parent update belongs to the later parent-child create test cases, not this minimum root-create case.

Current code behavior:
FacilitiesService.createFacility(...)
For parent_id = "" or null:
- Inserts one new row into facilities_tree.
- Does not update a parent row because there is no parent.
- Does not delete any row.

Step 1: Before sending Postman request, run this SQL.
Record the count result.

SELECT COUNT(*) AS before_count
FROM facilities_tree;

Step 2: Send this minimum create request in Postman.

{
  "facility_name": "会議室テスト",
  "parent_id": "",
  "target_facility_id": ""
}

Step 3: From the Postman response, copy data.facility_id.
Example:

{
  "success": true,
  "error_code": null,
  "message": "ok",
  "data": {
    "facility_id": 1001
  }
}

Step 4: After sending Postman request, run this SQL.
Record the count result.

SELECT COUNT(*) AS after_count
FROM facilities_tree;

Passing condition for Insert/Delete:
after_count = before_count + 1

Step 5: Check that only the created row has the same created_at and updated_at timestamp.
Replace <response_facility_id> with the facility_id returned by Postman.

SELECT facility_id, created_at, updated_at, deleted_at
FROM facilities_tree
WHERE facility_id = <response_facility_id>;

Expected:
- The created row exists.
- created_at is the POST time.
- updated_at is the POST time.
- deleted_at is NULL.

Step 6: Check that no other row was updated at the same timestamp as the created row.
Replace <response_facility_id> with the facility_id returned by Postman.

SELECT other.facility_id, other.updated_at
FROM facilities_tree created
JOIN facilities_tree other
  ON other.updated_at = created.updated_at
WHERE created.facility_id = <response_facility_id>
  AND other.facility_id <> created.facility_id;

Expected SQL result:
0 rows.

Passing condition:
- facilities_tree count increased by exactly 1.
- The created facility row exists.
- No other row has the same updated_at timestamp as the created row.

Failing condition:
- Count increased by more than 1.
- Count did not increase.
- Any other row has the same updated_at timestamp.
- Any unrelated row was deleted.

14-4 sort_order target_facility_id指定時の計算値
------------------------------------------------

Description:
The new sort_order is calculated from the target facility and the next facility under the same parent.

Meaning:
First, get the target facility sort_order. This is value ①.
Second, get the next facility sort_order under the same parent. This is value ②.
If ② does not exist, the new facility is placed at the end.
If ② exists, the new facility is placed between ① and ②.

SQL for ①:

SELECT sort_order
FROM facilities_tree
WHERE parent_id = <requested_parent_id>
  AND facility_id = <target_facility_id>
  AND deleted_at IS NULL;

SQL for ②:

SELECT sort_order
FROM facilities_tree
WHERE parent_id = <requested_parent_id>
  AND facility_id > <target_facility_id>
  AND deleted_at IS NULL
ORDER BY facility_id ASC
LIMIT 1;

Expected:
- If ② does not exist: new sort_order = ① sort_order + 10000000.
- If ② exists: new sort_order = (① sort_order + ② sort_order) / 2.

Example:
If ① = 10000000 and ② does not exist, new sort_order = 20000000.
If ① = 10000000 and ② = 20000000, new sort_order = 15000000.

Test:
Confirm the created facility sort_order equals the expected calculated value.

Troubleshooting if Postman returns 500:

If the error says INSERT INTO map_points failed, first check whether the map_points table exists and has the required columns.

Check columns:

SELECT column_name, data_type
FROM information_schema.columns
WHERE table_name = 'map_points'
ORDER BY ordinal_position;

If this returns no rows, create the table:

CREATE TABLE IF NOT EXISTS map_points (
  map_point_id BIGSERIAL PRIMARY KEY,
  map_id BIGINT NOT NULL REFERENCES maps(map_id),
  facility_id BIGINT NOT NULL REFERENCES facilities_tree(facility_id),
  x DOUBLE PRECISION NOT NULL,
  y DOUBLE PRECISION NOT NULL,
  target_type VARCHAR(128) NOT NULL,
  target_id BIGINT NOT NULL REFERENCES facilities_tree(facility_id),
  created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
  deleted_at TIMESTAMPTZ
);

Then send the same Postman request again.
